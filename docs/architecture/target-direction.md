# Policy Boundary Contracts

KiriView uses coherent Rust policy units rather than many small FFI helpers. A Rust module represents a meaningful workflow or algorithmic domain, such as opening, navigation, deletion, zoom, spread layout, display bucket selection, parsing, or cache policy.

Small local branches stay in C++ unless they are part of a larger Qt-independent policy decision. Moving a branch to Rust is appropriate when it clarifies ownership, strengthens tests, or lets multiple C++ runtime paths share one policy result. It is inappropriate when the bridge types and conversions become larger than the decision being moved.

Policy boundaries are fewer and larger:

- Workflow reducers that accept plain events and return state deltas plus effects.
- Geometry and rendering policy modules that compute values without owning renderer objects.
- Parsing modules that inspect bytes and return plain metadata or decoded domain values.
- Cache prioritization and navigation policies that can be tested without Qt event loops and do not own Qt runtime objects.

Document-session language-boundary changes cross coherent contracts rather than individual property-shaped helpers. Stable boundaries include active navigation projection from session and image-document snapshots, direct media routing plans, ordinary direct media deletion fallback plans, action-availability projections, and a named still-image-only direct media predecode eligibility policy.

Session-to-leaf document ports expose cohesive snapshot families and one snapshot-change event per family, not property-shaped signal bags. Leaf snapshot ports stay separate from leaf command/effect ports: session projection refresh and thumbnail-source adaptation consume `DocumentSessionImageDocumentSnapshot` and `DocumentSessionVideoDocumentSnapshot` values, while routing, page navigation, deletion, and stop cross explicit command/effect ports. Image source assignment, image-page navigation, and image-document deletion effects cross a focused session image-document command runtime through grouped source, page-navigation, and deletion subports so removing one feature does not require auditing the whole image command surface. Video source assignment, playback stop, and video-output access/attachment effects cross a focused session video-document command runtime through grouped source, playback, and output subports; accepted video-output attachment effects still cross a focused video-output runtime attachment port after the session runtime has applied public-projection eligibility.

Active-navigation dispatch is a named document-session subowner. The subowner may hold pending reveal context and execute typed direct-media or image-page navigation operations through narrow ports, while `DocumentSessionRuntime` supplies the current public active-navigation source/snapshot and `DocumentSessionState` remains the only public projection owner.

Session projection publication is a named document-session subowner. The projection runtime applies revisioned `DocumentSessionPublicSnapshotInput` values to `DocumentSessionState`, publishes image-page-only active-navigation updates through the same state owner, and synchronizes active-navigation thumbnail rows from the committed projection facts. `DocumentSessionRuntime` supplies snapshot inputs and leaf thumbnail rows, but it must not own the snapshot commit algorithm or thumbnail-row publication.

Session public-snapshot input assembly is a named document-session subowner. The input builder constructs `DocumentSessionPublicSnapshotInput` from committed session state, cached image/video leaf snapshots, direct-media navigation facts, and operation-availability facts. A session public-snapshot input port owns monotonically increasing input revision assignment and binds those committed fact sources to the builder. `DocumentSessionRuntime` supplies cached leaf snapshots and state ports to that input port, but it must not inline the projection-input field mapping, maintain its own public-snapshot input revision counter, or query leaf facades while building an input.

Session public leaf-snapshot assembly is a named document-session subowner. The leaf snapshot builder translates cohesive `DocumentSessionImageDocumentSnapshot` and `DocumentSessionVideoDocumentSnapshot` values into `DocumentSessionPublicImageLeafSnapshot` and `DocumentSessionPublicVideoLeafSnapshot` values. `DocumentSessionRuntime` may cache the built public leaf snapshots, but it must not inline raw leaf-to-public field mapping.

Direct-media deletion completion application is a named document-session subowner. The completion runtime clears deletion progress, reports typed file-operation failures through `DocumentSessionState`, and forwards accepted fallback route plans to the route runtime. `DocumentSessionRuntime` supplies state and route ports, but it must not inline completion-result branching.

Route planning and execution belong to a named document-session subowner. The route runtime classifies source/media URLs into route plans, owns route-operation dispatch, and keeps route-local execution facts such as direct-media scope changes. Route plan data distinguishes ordered mutation operations from typed follow-up effect requests instead of expressing publication, direct-media refresh, or predecode clearing as peers of state mutation. `DocumentSessionRuntime` supplies the current document kind plus typed mutation/effect ports for leaf routing, public projection publication, direct-media navigation refresh, predecode clearing, and route-completion cleanup. Route execution collects mutation facts first, then publishes accepted public projection changes once, then runs follow-up effects such as direct-media navigation refresh and predecode clearing. The route runtime port boundary groups session lifecycle, direct-media, leaf-document, source-identity, and follow-up callbacks instead of exposing one flat bag of unrelated callbacks.

Direct-image cursor synchronization is a named document-session planner. The planner decides whether a pending direct-image cursor is confirmed, restored after failure, or left unchanged from committed cursor and image leaf facts. The image synchronization application subowner applies the returned cursor operation to `DocumentSessionState`, but runtime adapters must not inline pending/displayed/source/error branching.

Image document synchronization is split between named document-session planners and a named session application subowner. The planners decide, from committed session facts, image leaf snapshots, and the direct-image cursor sync result, whether an image leaf change updates session source identity, mirrors image deletion progress, refreshes or caches direct-media navigation, publishes image-page active navigation, or recomputes the public projection. The application subowner applies the returned operations through state, direct-media navigation, predecode, reveal, and projection ports. `DocumentSessionRuntime` may refresh the cached public image leaf snapshot and invoke that subowner, but it must not inline cursor, direct-media, deletion-progress, or projection effect sequencing.

Video document synchronization is split between a named document-session planner and a named session application subowner. The planner decides whether a video leaf snapshot leaves the session unchanged, clears session direct-media state for an empty source, or commits a direct-video cursor from the leaf source. The application subowner applies the returned operation through state, direct-media navigation, projection, and active-zoom ports. `DocumentSessionRuntime` may refresh the cached public video leaf snapshot and invoke that subowner, but it must not inline empty-source/direct-video branching or the follow-up effect sequence.

Direct-media navigation application is a named document-session subowner. The application runtime translates inactive-refresh clearing and refresh/open workflow applications into state replacement, reveal-context operations, projection publication, predecode clearing or scheduling, and follow-up media routes through narrow ports. `DocumentSessionRuntime` may supply current cursor and active-navigation facts, but it must not inline workflow-application branching or own the effect sequence.

Direct-media navigation command coordination is a named document-session subowner. The coordinator owns refresh/open eligibility checks, current-scope reads, stale-completion acceptance, active-cursor reads, and callback forwarding between the async candidate runtime and the application runtime. `DocumentSessionRuntime` may bind the coordinator to session state, direct-media scope/activity ports, and projection/predecode/route effects, but it must not inline the direct-media refresh/open adapter sequence.

Viewport scan-start state, pan/scan command planning, anchored zoom positioning, viewport point-query geometry, and viewport command projection application belong to the image-document runtime/presentation layer, not QML. The facade may keep forwarding QML commands and a non-rendering QQuickItem bridge may adapt the physical `Flickable` position, but C++ owns command revision tracking, begin/complete/acknowledge ordering, and observation revision reconciliation.

Those boundaries keep `DocumentSessionState` as the public C++ owner and keep Qt/KDE effects in C++. Rust hosts pure projection or plan computation only from plain snapshots; C++ applies the resulting state, executes document routing, publishes signal batches, and preserves cursor-generation rejection for stale async completions.

When adding C++ or Rust files for these boundaries, keep source ownership manifests in sync, including `src/rust_policy_sources.txt`, `src/rust_bridge_sources.txt`, and relevant REUSE coverage.

C++ remains the place where those plans are executed through Qt/KDE APIs.
