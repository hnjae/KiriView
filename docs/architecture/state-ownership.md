# State Ownership

For QObject-facing workflows, C++ owns the authoritative runtime state. This includes QML-facing properties, Qt notification ordering, `QUrl`, `QImage`, `QString`, async job lifetime, presentation objects, and rendering objects.

Rust reducers operate on value snapshots and plain events. They return explicit state deltas, transition plans, and effect descriptions. Those results describe what C++ should apply; they are not an independent authoritative copy of the same workflow state.

Rust-owned state is reserved for self-contained Qt-independent domains where the state can be represented as plain values and does not mirror authoritative C++ state. Examples include format parsing state and geometry or zoom algorithms. Navigation indices, cache policy state, or other workflow state may move to Rust only when that ownership is documented in this file or an ADR and exposed through value-based FFI.

Avoid split-brain state. A workflow value must have one canonical owner. If both languages need to observe it, one side owns the value and the other side receives a derived snapshot, projection, delta, or completion event.

## Current Cross-Language Ownership

These defaults apply until this document or an ADR explicitly changes the owner. Moving policy into Rust does not move the authoritative runtime state unless the same decision also names the new state owner.

- Actions and shortcuts: C++ owns action identity, configurable shortcut stores, shortcut persistence, runtime-owned `QAction` enabled and checked state, menu and toolbar action text, shortcut display text, and the active shortcut handling installed in Qt. Configurable program-wide action shortcuts belong to the KDE/Kirigami action collection. Configurable viewer-local shortcuts belong to a KiriView-owned shortcut settings/model/router that keeps bare viewer keys out of ordinary `QAction::shortcuts()`. Command shortcuts belong to the action collection only when their declared activation scope is program-wide; non-command local input shortcuts that interpret a focused editor, dialog, or viewport gesture belong to the relevant QML item or local handler. QML reports UI-local gate facts such as focused text input, help-dialog visibility, and placement-only panel/fullscreen state through explicit inputs, then presents and invokes C++ action proxies without writing shared `QAction` state, deciding shortcut scope, or duplicating viewer command handling. Rust may compute stateless action and shortcut availability projections, such as menu-safe representatives, scope gates, or active-media shortcut enablement, but it must not store shortcut state.
- Application menu presentation: the C++ application runtime owns the live public menu-presentation value exposed to QML. KConfig stores the persisted snapshot, and the show-menubar `QAction` mirrors the runtime-owned value instead of becoming a second owner.
- Document and image loading: C++ owns the current document kind, displayed URL or image-document page, pending load identity, active jobs, decoded image objects, and user-visible load/error state. Primary image load failures cross the document load boundary as typed values with source URL, load session id, failure kind, user-facing text, diagnostic detail, severity, and retryability rather than as raw strings. `ImageDocumentState` publishes only the typed user-facing text to QML; decoder, data-loader, and presentation details remain diagnostic payloads for logs or tests. Rust may compute transition plans from a snapshot, but it must not keep a second displayed-image or pending-selection state.
- Document sessions: `DocumentSessionState` remains the C++ owner of the public mixed-media session state, including source identity, active document kind, direct media cursor, active zoom readout, active navigation projection inputs, file-deletion progress, displayed-media Open With availability, session-scoped error text, and public signal batches. Runtime coordinators may route image/video documents, execute jobs, launch displayed-media Open With requests, claim video-output surfaces, and schedule predecode work, but they publish those public values through this session-state owner. Displayed-media Open With target selection remains a session projection decision, while the provider invocation, cancellation, and stale-completion rejection belong to a focused session Open With runtime. Direct-media deletion candidate lookup, KDE deletion invocation, cancellation, and stale-completion rejection belong to a focused session deletion runtime; the session runtime may still gate the command on the current public projection, publish deletion progress through `DocumentSessionState`, and apply the completion route. Direct-media navigation candidate loading, cancellation, and stale-completion rejection belong to the direct-media navigation runtime; interpreting refresh/open completions into navigation state replacement, reveal-context decisions, predecode scheduling, and follow-up media routes belongs to a named direct-media navigation workflow helper before the session runtime applies those effects to `DocumentSessionState`. Video-output surface token issuance, owner freshness, detach authorization, clear-on-mode-exit state, and video-output attachment/detach effects belong to a focused session video-output runtime; the session runtime may still gate claims on the current public projection and active document kind before delegating the accepted claim to that runtime. Leaf image and video document facades may expose their current `sourceUrl` as a readonly observation, but top-level media selection must enter through `KiriDocumentSession`; production routing into leaf documents is allowed only through session-owned internal ports. Session-facing leaf reads must cross as cohesive image/video session snapshots through snapshot ports that are separate from command/effect ports. Narrow leaf command/effect ports remain for setting source URLs, page navigation, deletion, and stopping video, while video-output attachment crosses the focused video-output runtime's attachment port. Public mixed-media values are committed as one revisioned session projection snapshot sampled from the session, image-document, video-document, and operation-availability leaf snapshots. The document-session runtime must keep a current cached leaf snapshot per input family and recompute public projections only from those cached values; it must not query arbitrary image/video facade getters while applying a public projection. A source, kind, error, active zoom, active navigation, title, deletion, Open With, or direct-media navigation completion change must publish the fully committed snapshot before emitting its derived signal batch, and QML-facing scalar getters read from that current snapshot rather than from independently recomputed leaf properties.
- Active navigation projection: the top-level C++ document session owns the public active navigation projection used by the main toolbar, shared actions, menus, shortcuts, and action availability. `KiriImageDocument` remains the canonical owner for image-document page navigation state, including spread-aware current first and last visible page, total count, previous and next availability, and first and last boundary state, but the document session consumes that full snapshot and republishes the active projection. Rust computes the pure projection and dispatch policy from plain session and image-document snapshots; C++ applies and publishes the result through `DocumentSessionState` and executes route operations. QML may render the projection and invoke session dispatch methods, but it must not recompute shared Previous, Next, First, Last, editability, boundary, or deletion-gated dispatch policy from raw image-document page numbers.
- Container navigation diagnostics: adjacent-container listing failures are diagnostic-only navigation outcomes. They must cross the navigation-plan and image-document runtime-plan boundary as typed diagnostic values that preserve current container URL, parent URL, direction, failure kind, and backend detail. Empty sibling lists, missing current containers, normal boundaries, and listing failures remain distinct internal outcomes even when the user-facing behavior is quiet.
- Navigation presentation ordering: one QML navigation presentation projection owns right-to-left-aware leading and trailing action slots, physical affordance icons, first and last icon projection, and ordered menu action arrays derived from the active action set and reading direction. Toolbar, menu bar, and application menu surfaces consume those slots; they must not map `rightToLeftReadingActive` directly to Previous, Next, First, or Last actions or icons.
- Active navigation thumbnail strip: the top-level C++ document session owns the QML-facing thumbnail-strip projection derived from the same confirmed active navigation list as the toolbar readout. A named document-session thumbnail subowner owns thumbnail runtime construction, session-specific source adaptation, thumbnail demand state, foreground and background fill scheduling, async job lifetime, queue priority, cancellation, stale-completion rejection, result status, debug observability, and `QImage` or image-provider storage for the active strip. `DocumentSessionRuntime` may publish committed active-navigation rows and delegate QML demand, but it must not own thumbnail source renderability or construct the lower-level thumbnail runtime directly. QML reports visible thumbnail demand and renders projected thumbnail or fallback results, but it must not schedule thumbnail work, resolve source renderability, or independently list sibling directories, archive entries, or image-document candidates. Source adapters are the boundary for source identity, renderability, cache identity candidates, and freshness facts. Background fill is internal to the C++ runtime: it may run only for the current active navigation generation after foreground visible or nearby demand has drained, it must be canceled or rejected on new foreground demand or scope changes, and it may fill the cache without publishing a lower-quality result over an existing ready row. The model projects results through source-neutral roles and statuses rather than direct-image-only ownership language, so unsupported, pending, failed, video, archive-entry, archive-collection, and directory-collection rows stay in the same fallback result path. Thumbnail runtime diagnostics must preserve job id, source key, work kind, demand bucket, failure kind, and error detail when cache lookup, generation, or image-store insertion fails, without expanding the QML-facing thumbnail model. The thumbnail image store is a C++ memory-pressure owner: it accounts stored `QImage` byte cost, retains currently visible demand ahead of nearby or background demand, evicts within a resolved thumbnail byte budget, and honors explicit release when a row result is replaced or discarded. Rust may compute the plain thumbnail cache budget from user request and system-memory snapshots, but it must not own the live `QImage` cache or eviction bookkeeping.
- Video source loading: C++ owns playback URL resolution, resolver cleanup, pending resolution identity, backend source clearing, and the public video source/status/error state. A dedicated source-load runtime may own the async resolver lifecycle, but `VideoDocumentState` remains the canonical owner for QML-facing source and status values. Source-load failures and backend playback failures should cross their runtime boundaries as typed values with source identity, failure kind, user-facing text, diagnostic detail, severity, and retryability rather than as raw backend strings. `VideoDocumentState` publishes only the typed user-facing text; backend, DBus, KIO, and Qt details remain diagnostic payloads for logs or tests.
- Image load sessions: the C++ image loader and session tracker own pending load identity, first-display decode context, image-page-scope context, and the resolved image URL for direct archive or directory opens. Page-scope candidate completion should update and return the accepted session snapshot; callbacks should not carry a second resolved URL beside the session-owned image URL.
- Embedded metadata parsing: Rust owns parsing image decode bytes and direct video local paths into plain embedded-metadata values. C++ owns when parsing is requested, which source backend supplies bytes or paths, stale-completion rejection, document-level storage, and the QML-facing `KiriMediaInformation` row models. Parser failures are empty metadata results, not document load failures.
- Supported image lists and page position: C++ owns the confirmed supported image list, current index, boundary state, pending page selection, refresh operation identity, and watched-source context for a runtime scope unless an ADR gives that scope a different owner. Rust navigation policy may return target indices or follow-up effects from those snapshots.
- Image presentation runtime: `ImagePresentationRuntime` is the single mutable owner of image presentation state exposed to QML. It owns current presentation mode, right-to-left reading, transition phase, zoom workflow state, rotation, logical viewport frame, visible source rect, scan-start handoff, page-slot visibility, provider display-source projections, restoration snapshots, and display refinement demand. Rust geometry or spread policy may calculate next values from snapshots, but it must not store a second live copy of those workflow values.
- Image page surface owners: page-level presentation objects are display-resource owners, not public presentation-state owners. They may retain animation readers, display entries, display-image pin leases, previous-frame retention, image revision, static-image payloads, predecode cacheability facts, and load lifetimes. They must not expose or store mutable zoom mode, zoom percent, rotation, logical viewport frame, visible source rect, page visibility, display-source policy, render-snapshot policy, tile caches, custom render resources, or public transition state. Legacy setters that directly write viewport content position, visible item rect, zoom percent, rotation, render-snapshot fields, or tile/render state are presentation-owner backdoors and must be removed rather than kept as compatibility helpers.
- Image viewport frame bridging: C++ owns the canonical logical viewport frame as a revisioned presentation projection. QML owns only Qt Quick item instances, physical `Flickable` content position, pointer anchors, gesture sampling, rebound, and animation state. Presentation commands publish monotonically increasing command revisions with requested content position, content size, image rect, visible item rect, pannability, outcome, and status; QML applies only the latest non-superseded command revision and reports actual physical state back as observations or acknowledgements with their own revision and origin. Gesture, inertia, overshoot, resize, rotation, device-pixel-ratio, and explicit acknowledgement observations may promote physical state into the canonical frame only through the C++ owner. Delayed QML callbacks must not write the logical frame or replay stale content positions.
- QML-facing stale-sensitive revisions are C++-owned. C++ may store revisions as `quint64`, but QML must not create them, store them in numeric `int` or JavaScript `number` properties, compare them arithmetically, or pass them through APIs that can narrow their value. QML may retain opaque token strings minted by the C++ owner and ask that owner whether a projection, claim, or acknowledgement is newer, applicable, superseded, rejected, or accepted.
- Image viewport commands: zoom step, manual zoom, fit mode, fit-or-actual-size toggling, keyboard pan, scan forward/backward, scan-start handoff, and content-position clamping are presentation commands. QML may pass UI-local anchors or deltas into those commands, but it must not calculate the resulting zoom percent, scan target, or canonical content position. The command owner records whether the command was applied, superseded, rejected, or promoted by an observation. Viewport command application should be driven by the command projection revision flow rather than by each individual QML command handler deciding when to apply the projection.
- Fit-mode presentation selection: the image presentation runtime owns the current fit-mode menu selection used by toolbar and menu checked state. Manual zoom may leave the last accepted fit-mode selection available for future fit commands, but QML must consume that runtime projection rather than storing a durable selected-fit-mode mirror or applying optimistic selection before a command is accepted.
- Media viewport composition: QML owns the shared media workspace host that composes the media viewport area with layout-reserving panels. The workspace host owns media-area SplitView layout, runtime-only panel visibility, panel sizing, and the public null-safe interaction surface consumed by shared shortcut wiring. Inside that workspace, the media viewport host selects and loads exactly one active media viewport delegate through the formal delegate contract. The viewport host manages delegate selection, shared viewport focus dispatch, overlay placement, and delegate signal forwarding, but it must call only delegate-contract APIs and must not reach into mode-specific viewport internals. Media viewport delegates own common viewport click and context-menu request forwarding through the shared contract. Image-specific local pan, zoom, spread, and high-level image/provider item instances are owned by the image viewport and facade/runtime stack and are exposed to shared wiring only through the image interaction surface; scan shortcut dispatch is an application command route, not an interaction-surface command. C++ owns display-source policy and display-entry lifetime. QML owns the `VideoOutput` item instance and may report a surface claim containing that item plus its visibility and geometry with a C++-minted opaque claim token, but backend attachment, detach, stale route rejection, destroyed-item weak handling, and public video-output state belong to the session/video runtime owner. Video-specific playback controls, seek handling, and Qt Multimedia integration remain owned by the video document/runtime stack and use the default no-op interaction surface for image-only commands.
- Application actions and shortcuts: the window-local C++ action runtime owns `QAction` instances, enabled state, checked state, shortcut routing, unsupported-video shortcut interception, repeated-trigger serialization, command routing, and revisions for accepted UI gate snapshots. QML owns placement and UI-local gates only. QML may report UI gate facts containing focus, modality, help-dialog, fullscreen chrome, and transient tool state, but it must not combine document, session, presentation, or viewport facts into shared action availability, manufacture action-state revisions, dispatch domain commands from action ids, instantiate shortcut navigation policy facades, decide whether a shortcut slot is program-wide or viewer-local, or duplicate viewer command handling. Shared image pannability gates are derived by C++ from the active image document viewport fact; QML may still use viewport-local pannability for local item interaction, but it must not pass that fact through the application UI gate snapshot. The application runtime combines UI-local gates with C++-owned document kind, readiness, projection, and viewport facts before enabling shared action or shortcut routes. Triggered-action dispatch and fixed viewer command routing, including image arrow/scan behavior and video seek dispatch, belong to an application command router that consumes owner-grouped shell, session, image-document, image-presentation, panel, window, help, and video-document ports rather than storing routing policy on the QML facade. Fixed video seek shortcuts are recognized by the shortcut runtime and dispatched through that router to the video document command path after the runtime applies its active mode and UI gate inputs.
- Animation playback: C++ owns animation reader lifetime, frame timer scheduling, source restart, loop progress, current frame delivery, and animation error reporting. Source-specific readers should expose plain frame/open results to the playback owner instead of duplicating timer or restart workflow decisions. The playback owner receives frame scheduling through a timer port so tests can fire frames without wall-clock waits while production keeps Qt timer behavior.
- View render context: the non-rendering `KiriImageViewportContextBridge` owns document render-context provider attachment, primary-only provider installation, component-completion inputs, scene-change invalidation, display DPR observation, and a conservative or public-API-derived texture cap. It must not draw, set `ItemHasContents`, implement `updatePaintNode()`, call `QQuickWindow::rhi()`, include QRhi headers, own textures, or schedule decode work. Secondary page visuals must not install or clear the document render context provider.
- Display refinement scheduling: the image presentation/page-surface owners translate visible provider-display projections into whole-image display bucket demands, display-source revisions, provider entries, and bounded-detail, unsupported, or failed states when one safe display image cannot represent the demand. Production image display must not schedule visual tiles, maintain decoded tile caches, project render frames, or retain scene-graph render resources.
- Location identity: the C++ location domain owns QUrl-based image, container, image-page-scope, displayed-image, directory-navigation, and navigation-source value objects plus normalized URL identity comparison. Runtime domains may store these values as their authoritative state or consume derived snapshots, but they should not invent parallel URL normalization or parent-directory derivation rules. Production navigation-source wrappers may collect platform facts such as document-portal host paths and KIOFuse runtime directories, but the identity derivation itself should be testable from explicit facts.
- Deletion and Open With file operations: Rust policy may choose the deletion target and post-delete follow-up target from plain document/navigation snapshots. C++ owns KDE confirmation, the shared file-deletion and Open With side-effect providers, cancellation and failure handling, notifications, and applying the follow-up plan. Direct-media deletion fallback is a session-owned plan, but the deletion runtime owns the candidate-loading lifecycle used to build that fallback and must reject stale candidate or file-operation completions before returning a completion to the session. KDE/KIO operation failures cross those provider boundaries as typed values with operation kind, target URL, raw KJob/KIO error code when available, cancellation status, user-facing text, diagnostic detail, and retryability rather than as raw strings. Removal fallback for an image-document page or comic-book container disappearing from its candidate set is a navigation policy consumed by both live candidate refresh and image-document deletion. Ordinary direct media deletion fallback remains document-session owned.
- Preparation and cache: C++ owns prepared image objects, decoder jobs, display image entries, memory pressure handling, and cache lifetime. Rust may compute preparation priority or eviction policy from plain metadata, but cached runtime objects remain in C++.
- Ordinary direct media predecode: the C++ document-session media-predecode runtime owns the predecode/cache lifetime for mixed image/video direct media scopes. It owns the media predecode coordinator, schedule/cache eligibility checks, cancellation, clear behavior, cached predecoded-image lookup, and the conversion of committed session facts into coordinator schedule context. `DocumentSessionRuntime` may supply the current direct-media activity flag, cursor URL, active image/displayed predecode facts, and first-display decode context from its committed session and leaf snapshots, but it must not own the predecode coordinator lifecycle or duplicate schedule/cache eligibility. Session-owned direct media predecode has its own decode, power-saver, timer, thread-count, and cache-budget dependency boundary instead of borrowing image-document runtime dependency ownership. Shared predecode scheduling runtime state, including debounce timers, power-saver reaction, generation acceptance, pending work context, and dispatch of schedule operations, is owned by the C++ predecode runtime. Image and mixed-media coordinators provide only their domain-specific candidate and window planning, receive scheduler/clock and host-thread facts through explicit dependency inputs, and do not probe wall-clock timers or system thread count directly. Any candidate snapshot needed after debounce or power-saver resume must travel inside the pending schedule context instead of beside it. Direct media predecode eligibility is still-image-only: video items may be used as cursor positions for image-predecode priority, but cached runtime objects remain provider-ready still-image display payloads and video frame predecode is not in scope.
- HEIF source-internal tiling: C++ owns libheif context and image handles, QImage allocation, and painting requested source rects into one display image while Rust owns tile-grid validation and source-rect tile-region planning from plain numeric tiling metadata. Source-internal HEIF tiling is allowed only inside a decoder or refinement job that assembles one accepted display `QImage`; it must not publish multiple visual page tiles or reintroduce viewport tile scheduling.
- System runtime state: C++ owns desktop-environment observer lifetime and platform probes, including power-saver portal subscription state and physical-memory discovery. Memory-sensitive runtime owners should consume a C++ system-memory snapshot instead of probing physical memory independently. Rust may compute cache or scheduling policy from plain snapshots derived from those observers.
- Public projection owners: the document session owns named immutable projections for active navigation, active zoom, window-title subject, displayed-media operation availability, media information, active-navigation thumbnails, shared UI control facts, active video-control readiness, and any session-fed action-availability facts used by application actions. Each owner consumes one committed session snapshot or leaf-snapshot input revision, replaces its current C++ snapshot, and exposes only stable QObject facades or scalar getters to QML. Projection owners may call pure helpers, but while applying a snapshot they must not query arbitrary leaf facades or other projection owners for missing policy inputs. QML may use the published facts for placement, but it must not derive shared menu ordering, toolbar control visibility, active media readiness, or action availability from leaf document properties.
- Projection update rule: public projection state advances only by replacing a revisioned snapshot. Partial public projection APIs that update active navigation, title, media information, operation availability, shared control facts, active video readiness, or active zoom independently are backdoors and must be removed after their replacement tests exist.
- Compatibility boundary rule: once a durable public value has a named owner, production facades must not keep public mutators, writable QML properties, direct event-filter dispatch, or helper APIs that can bypass that owner. Test fixtures may drive an owner through a dedicated fixture port, but production C++ callers must use the same typed command or snapshot-fact API that QML uses.
- UI gate rule: UI-local action gates are facts, not policy. QML may report gate facts for focus, modality, active or hovered view, fullscreen chrome, and transient tools. The action runtime assigns the accepted gate snapshot revision and must never recompute shared action availability from unrevisioned QML booleans.
- Action proxy rule: QML action proxies are placement adapters only. They must not expose `enabled`, `checked`, or `checkable` override properties for runtime-owned commands. Placement availability may hide or omit a proxy, but cannot rewrite the QAction state published by the application action runtime.
- Fixed shortcut rule: fixed viewer-only shortcuts are runtime shortcut routes. They are not user-configurable actions, but they share the action runtime's newest UI gate, modal suppression, focused-view selection, trigger serialization, and command dispatch path. QML viewport handlers may capture Qt-local gestures, but they must not duplicate fixed viewer command shortcuts such as image pan/navigation or video seek semantics.

## Minimum Source Key Contract

The C++ location and session domains own source identity keys for route sealing, route equality, route freshness, session snapshots, and direct-media scope identity.

Top-level route keys normalize `QUrl` path segments, preserve query and fragment components, and compare fully encoded identity strings. They do not case-fold and do not resolve symlinks, aliases, shortcuts, or platform filesystem equivalences.

Local file keys additionally clean local path syntax and make relative local paths absolute for identity only. The public `sourceUrl` remains the requested URL rather than the identity URL.

Direct-media scope identity is `{ current key, parent key, generation }`. The current and parent keys use the existing navigation-source rules, including document-portal host paths and KIOFuse archive restoration, before applying source-key normalization. Platform probes for those navigation-source rules are adapter work; pure identity helpers consume resolved facts and do not read xattrs or process environment directly.

Direct-media freshness changes only when the effective current key or parent scope key changes. Resolving pending direct-image confirmation to an equivalent displayed URL is a phase change within the same freshness generation.

Archive-entry key families, render keys, thumbnail cache keys, predecode candidate keys, and sandbox-specific freshness beyond the existing navigation-source handling are defined by the extension contracts. Code that crosses adapter, cache, thumbnail, predecode, or render boundaries should use typed family-specific keys instead of a generic URL identity key.

The generic top-level source key is not an adapter boundary type. Extension boundaries must use family-specific key structs whose equality cannot accidentally compare unrelated families. Durable identity and freshness generation are separate fields; a generation change accepts or rejects work for the current owner scope, while equality compares only durable identity inside one key family.

## Derived Public State

QML-facing values may be derived from multiple C++ runtime states. For example, a public loading or status property may combine document state with an active presentation transition. The derived value must not become a second mutable source of truth. Keep the canonical owners explicit, and make notification dependencies follow the derived value.

When a public value has mode-specific ownership, only the active mode owns that value. Inactive mode state is a cache, projection, or restoration point, not a competing owner. Transition code must synchronize the next active owner before exposing the mode change. For example, different presentation modes may use different internal state owners, but the public value must have exactly one active owner at a time.

Image presentation mode transitions publish an explicit phase: previous-active while the last committed presentation is still authoritative, transitioning-placeholder while no partial target presentation may be rendered, and committed-active after the new single-page or two-page spread presentation is complete. `ImagePresentationRuntime` holds the canonical zoom snapshot, rotation, viewport frame, visible source rect, spread mode, reading direction, primary and secondary visibility, and render projection for the committed mode. Preparing a secondary page may update loader-owned decoded surfaces, but it must not copy zoom into page-surface controllers, patch page visible rects, or expose a partially prepared spread as the active public owner. If a transition aborts, the runtime restores from the retained restoration snapshot and leaves the previous committed presentation authoritative.

Display-source projection and provider entries are the visual boundary for images. QML high-level image items consume accepted source facts while decode and refinement jobs publish whole display images. QML supplies viewport and scroll inputs only; it must not compute display buckets, SVG scale buckets, provider ids, refinement demands, stale-display fallback policy, or cache retention. Window changes, scene-graph invalidation, DPR changes, and texture-capability changes advance render-context freshness for display-bucket demand rejection, not for app-owned QRhi resources.

The active navigation projection must clear or mark unknown values during empty startup, loading intervals without a confirmed same-scope selection, mode switches, and replacement failures before publishing any current/count pair. Unknown navigation disables entry and shared dispatch, and must not display stale current/count values from a previous document.

Shared Previous and Next requests return their dispatch or boundary outcome from the document-session owner. The action runtime owns visible placement availability for Previous and Next from the session active-navigation projection, including disabled first/last boundary placements, while shortcut and fixed viewer-navigation paths may still ask the session for boundary feedback. QML may turn a first/last boundary outcome into user-facing feedback, but it must not decide boundary-vs-dispatch from raw current/count or mode-specific document values.

The main window title subject is a document-session-owned derived public value. Leaf image and video documents may own file-name-only title inputs and intrinsic media metadata, and image documents own page-navigation state, but the session combines those inputs into the public title subject. QML may append the application name around that subject, but it must not recompute direct-media size, document page counters, or mode-specific title fallback policy from raw document properties.

Media information is a document-session-owned projection. Leaf image and video documents own raw displayed URL, readiness, opened-collection scope, dimensions, and embedded metadata snapshots, but the session chooses the current information target, title, summary, row groups, copy-path availability, and containing-folder availability from the same committed mixed-media input revision as the rest of the public session projection. `KiriMediaInformation` is a stable QML facade over that snapshot and owns only Qt model notification mechanics plus clipboard and file-manager side effects.

Direct media sibling discovery may complete asynchronously, but the cursor used for the eventual public readout is the session-owned requested direct media URL for that open request. It must not be derived from a stale or empty image-document displayed URL. The session-owned direct media scope snapshot contains the effective cursor URL, parent scope URL, and generation used to accept or reject sibling-list completions; pending-image-to-stable-image confirmation is a phase change, not a new scope, when those URLs are unchanged.

Action availability is a session-fed named projection consumed by the application action runtime. It may include image-internal, viewport, shortcut-gating, and presentation-only facts, but those facts must enter the action runtime as C++ snapshots rather than QML-computed booleans. Shared Previous, Next, First, and Last availability comes from the session active navigation projection.

Extracting active navigation helpers must not move authoritative state to Rust. Rust may later compute pure projections from plain snapshots, but C++ applies and publishes the result through the document session.

Direct media cursor generation is part of stale async-completion rejection. Future helper extraction must preserve idempotent cursor operations and reject candidate results unless the cursor URL, scope URL, and generation still match the accepted session request. Cursor generation changes when the effective direct media identity or parent scope changes, not merely when ownership transitions from pending to confirmed for the same direct media URL.

Mixed-media deletion fallback remains session-owned for ordinary direct media scopes. Image-document removal fallback remains navigation-owned so deletion completion and live candidate refresh share one sibling-selection policy.
