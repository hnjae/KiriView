# State Ownership

For QObject-facing workflows, C++ owns the authoritative runtime state. This includes QML-facing properties, Qt notification ordering, `QUrl`, `QImage`, `QString`, async job lifetime, presentation objects, and rendering objects.

Rust reducers operate on value snapshots and plain events. They return explicit state deltas, transition plans, and effect descriptions. Those results describe what C++ should apply; they are not an independent authoritative copy of the same workflow state.

Rust-owned state is reserved for self-contained Qt-independent domains where the state can be represented as plain values and does not mirror authoritative C++ state. Examples include format parsing state and geometry or zoom algorithms. Navigation indices, cache policy state, or other workflow state may move to Rust only when that ownership is documented in this file or an ADR and exposed through value-based FFI.

Avoid split-brain state. A workflow value must have one canonical owner. If both languages need to observe it, one side owns the value and the other side receives a derived snapshot, projection, delta, or completion event.

## Current Cross-Language Ownership

These defaults apply until this document or an ADR explicitly changes the owner. Moving policy into Rust does not move the authoritative runtime state unless the same decision also names the new state owner.

- Actions and shortcuts: C++ owns action identity, KDE action collections, the canonical configured shortcut list, shortcut persistence, runtime-owned `QAction` enabled and checked state, menu and toolbar action text, shortcut display text, and the active shortcut handling installed in Qt. Command shortcuts belong to the action collection even when they are fixed and user-nonconfigurable; non-command local input shortcuts that interpret a focused editor, dialog, or viewport gesture belong to the relevant QML item or local handler. QML reports UI-local gate facts such as focused text input, help-dialog visibility, and placement-only panel/fullscreen state through explicit inputs, then presents and invokes C++ action proxies without writing shared `QAction` state. Rust may compute stateless action and shortcut availability projections, such as menu-safe representatives, derived viewer aliases, scope gates, or active-media shortcut enablement, but it must not store shortcut state.
- Application menu presentation: the C++ application runtime owns the live public menu-presentation value exposed to QML. KConfig stores the persisted snapshot, and the show-menubar `QAction` mirrors the runtime-owned value instead of becoming a second owner.
- Document and image loading: C++ owns the current document kind, displayed URL or image-document page, pending load identity, active jobs, decoded image objects, and user-visible load/error state. Rust may compute transition plans from a snapshot, but it must not keep a second displayed-image or pending-selection state.
- Document sessions: `DocumentSessionState` remains the C++ owner of the public mixed-media session state, including source identity, active document kind, direct media cursor, active zoom readout, active navigation projection inputs, file-deletion progress, displayed-media Open With availability, session-scoped error text, and public signal batches. Runtime coordinators may route image/video documents, execute jobs, launch displayed-media Open With requests, and schedule predecode work, but they publish those public values through this session-state owner. Leaf image and video document facades may expose their current `sourceUrl` as a readonly observation, but top-level media selection must enter through `KiriDocumentSession`; production routing into leaf documents is allowed only through session-owned internal ports. Public mixed-media values are committed as one revisioned session projection snapshot sampled from the session, image-document, video-document, and operation-availability leaf snapshots. A source, kind, error, active zoom, active navigation, title, deletion, or Open With change must publish the fully committed snapshot before emitting its derived signal batch, and QML-facing scalar getters read from that current snapshot rather than from independently recomputed leaf properties.
- Active navigation projection: the top-level C++ document session owns the public active navigation projection used by the main toolbar, shared actions, menus, shortcuts, and action availability. `KiriImageDocument` remains the canonical owner for image-document page navigation state, including spread-aware current first and last visible page, total count, previous and next availability, and first and last boundary state, but the document session consumes that full snapshot and republishes the active projection. Rust computes the pure projection and dispatch policy from plain session and image-document snapshots; C++ applies and publishes the result through `DocumentSessionState` and executes route operations. QML may render the projection and invoke session dispatch methods, but it must not recompute shared Previous, Next, First, Last, editability, boundary, or deletion-gated dispatch policy from raw image-document page numbers.
- Active navigation thumbnail strip: the top-level C++ document session owns the QML-facing thumbnail-strip projection derived from the same confirmed active navigation list as the toolbar readout. A document-session-owned C++ thumbnail runtime owns thumbnail demand state, foreground and background fill scheduling, async job lifetime, queue priority, cancellation, stale-completion rejection, result status, debug observability, and `QImage` or image-provider storage for the active strip. QML reports visible thumbnail demand and renders projected thumbnail or fallback results, but it must not schedule thumbnail work, resolve source renderability, or independently list sibling directories, archive entries, or image-document candidates. Source adapters are the boundary for source identity, renderability, cache identity candidates, and freshness facts. Background fill is internal to the C++ runtime: it may run only for the current active navigation generation after foreground visible or nearby demand has drained, it must be canceled or rejected on new foreground demand or scope changes, and it may fill the cache without publishing a lower-quality result over an existing ready row. The model projects results through source-neutral roles and statuses rather than direct-image-only ownership language, so unsupported, pending, failed, video, archive-entry, archive-collection, and directory-collection rows stay in the same fallback result path. The thumbnail image store is a C++ memory-pressure owner: it accounts stored `QImage` byte cost, retains currently visible demand ahead of nearby or background demand, evicts within a resolved thumbnail byte budget, and honors explicit release when a row result is replaced or discarded. Rust may compute the plain thumbnail cache budget from user request and system-memory snapshots, but it must not own the live `QImage` cache or eviction bookkeeping.
- Video source loading: C++ owns playback URL resolution, resolver cleanup, pending resolution identity, backend source clearing, and the public video source/status/error state. A dedicated source-load runtime may own the async resolver lifecycle, but `VideoDocumentState` remains the canonical owner for QML-facing source and status values.
- Image load sessions: the C++ image loader and session tracker own pending load identity, first-display decode context, image-page-scope context, and the resolved image URL for direct archive or directory opens. Page-scope candidate completion should update and return the accepted session snapshot; callbacks should not carry a second resolved URL beside the session-owned image URL.
- Embedded metadata parsing: Rust owns parsing image decode bytes and direct video local paths into plain embedded-metadata values. C++ owns when parsing is requested, which source backend supplies bytes or paths, stale-completion rejection, document-level storage, and the QML-facing `KiriMediaInformation` row models. Parser failures are empty metadata results, not document load failures.
- Supported image lists and page position: C++ owns the confirmed supported image list, current index, boundary state, pending page selection, refresh operation identity, and watched-source context for a runtime scope unless an ADR gives that scope a different owner. Rust navigation policy may return target indices or follow-up effects from those snapshots.
- Zoom, pan, rotation, scan-start handoff, and spread presentation: the active C++ presentation owner owns the public presentation state exposed to QML. Rust geometry or spread policy may calculate next values from snapshots. Single-page and two-page spread internals may retain restoration snapshots, displayed surfaces, decode jobs, or page-local render resources, but inactive mode state must not be a second mutable owner of active zoom, rotation, viewport frame, public transition state, or active render projection.
- Image viewport frame bridging: C++ owns the canonical logical viewport frame as a revisioned presentation projection. QML owns only Qt Quick item instances, physical `Flickable` content position, pointer anchors, gesture sampling, rebound, and animation state. Presentation commands publish monotonically increasing command revisions with requested content position, content size, image rect, visible item rect, pannability, outcome, and status; QML applies only the latest non-superseded command revision and reports actual physical state back as observations or acknowledgements with their own revision and origin. Gesture, inertia, overshoot, resize, rotation, device-pixel-ratio, and explicit acknowledgement observations may promote physical state into the canonical frame only through the C++ owner. Delayed QML callbacks must not write the logical frame or replay stale content positions.
- Image viewport commands: zoom step, manual zoom, fit mode, fit-or-actual-size toggling, keyboard pan, scan forward/backward, scan-start handoff, and content-position clamping are presentation commands. QML may pass UI-local anchors or deltas into those commands, but it must not calculate the resulting zoom percent, scan target, or canonical content position. The command owner records whether the command was applied, superseded, rejected, or promoted by an observation.
- Media viewport composition: QML owns the shared media workspace host that composes the media viewport area with layout-reserving panels. The workspace host owns media-area SplitView layout, runtime-only panel visibility, panel sizing, and the public null-safe interaction surface consumed by shared shortcut wiring. Inside that workspace, the media viewport host selects and loads exactly one active media viewport delegate through the formal delegate contract. The viewport host manages delegate selection, shared viewport focus dispatch, overlay placement, and delegate signal forwarding, but it must call only delegate-contract APIs and must not reach into mode-specific viewport internals. Media viewport delegates own common viewport click and context-menu request forwarding through the shared contract. Image-specific rendering, pan, zoom, scan, spread, and QRhi scene-graph work remain owned by the image viewport and facade/runtime stack and are exposed to shared wiring only through the image interaction surface. Video-specific `VideoOutput` attachment, playback controls, seek handling, and Qt Multimedia integration remain owned by the video viewport and video document/runtime stack and use the default no-op interaction surface for image-only commands.
- Application actions and shortcuts: the window-local C++ action runtime owns `QAction` instances, enabled state, checked state, shortcut routing, unsupported-video shortcut interception, repeated-trigger serialization, and command routing. QML owns placement and UI-local gates only. QML may report a revisioned UI gate snapshot containing focus, modality, help-dialog, fullscreen chrome, and transient tool state; it must not combine document, session, presentation, or viewport facts into shared action availability, and it must not dispatch domain commands from action ids. Command routing from a triggered action goes through the C++ action runtime to the owning session, document, presentation, or shell-side-effect callback.
- Animation playback: C++ owns animation reader lifetime, timer scheduling, source restart, loop progress, current frame delivery, and animation error reporting. Source-specific readers should expose plain frame/open results to the playback owner instead of duplicating timer or restart workflow decisions.
- View render context: `KiriImageView` owns render-context discovery for the attached document only when it is the primary viewport facade. The view facade owns document attachment, page role, and component-completion inputs; render-context binding helpers should consume those values as snapshots and only track whether a provider is currently installed. Secondary page views render secondary snapshots but do not install or clear the document render context provider.
- Tile decode scheduling and render-frame projection: the C++ rendering runtime owns the active displayed-surface generation, pending tile keys, failed tile keys, worker completion acceptance, current frame projection, and frame invalidation for visible static-image rendering. Tile selection helpers may compute request candidates, active tile layers, and draw-entry geometry from render-context and visibility snapshots, but they must not become a second owner of pending decode state or current-frame state.
- Location identity: the C++ location domain owns QUrl-based image, container, image-page-scope, displayed-image, and directory-navigation value objects plus normalized URL identity comparison. Runtime domains may store these values as their authoritative state or consume derived snapshots, but they should not invent parallel URL normalization or parent-directory derivation rules.
- Deletion: Rust policy may choose the deletion target and post-delete follow-up target from plain document/navigation snapshots. C++ owns KDE confirmation, the shared file-deletion side-effect provider, cancellation and failure handling, notifications, and applying the follow-up plan. Image-document deletion fallback remains image-document owned, and ordinary direct media deletion fallback remains document-session owned.
- Preparation and cache: C++ owns prepared image objects, decoder jobs, memory pressure handling, and cache lifetime. Rust may compute preparation priority or eviction policy from plain metadata, but cached runtime objects remain in C++.
- Ordinary direct media predecode: the C++ document session owns the predecode/cache lifetime for mixed image/video direct media scopes. Session-owned direct media predecode has its own decode, power-saver, and cache-budget dependency boundary instead of borrowing image-document runtime dependency ownership. Shared predecode scheduling runtime state, including debounce timers, power-saver reaction, generation acceptance, pending work context, and dispatch of schedule operations, is owned by the C++ predecode runtime. Image and mixed-media coordinators provide only their domain-specific candidate and window planning, and any candidate snapshot needed after debounce or power-saver resume must travel inside the pending schedule context instead of beside it. Direct media predecode eligibility is still-image-only: video items may be used as cursor positions for image-predecode priority, but cached runtime objects remain still-image payloads and video frame predecode is not in scope.
- HEIF tiled rendering: C++ owns libheif context and image handles, QImage allocation, tile decoding, and painting into source-rect images. Rust owns tile-grid validation and source-rect tile-region planning from plain numeric tiling metadata.
- System runtime state: C++ owns desktop-environment observer lifetime and platform probes, including power-saver portal subscription state and physical-memory discovery. Memory-sensitive runtime owners should consume a C++ system-memory snapshot instead of probing physical memory independently. Rust may compute cache or scheduling policy from plain snapshots derived from those observers.
- Public projection owners: the document session owns named immutable projections for active navigation, active zoom, window-title subject, displayed-media operation availability, media information, active-navigation thumbnails, and any session-fed action-availability facts used by application actions. Each owner consumes one committed session snapshot or leaf-snapshot input revision, replaces its current C++ snapshot, and exposes only stable QObject facades or scalar getters to QML. Projection owners may call pure helpers, but while applying a snapshot they must not query arbitrary leaf facades or other projection owners for missing policy inputs.
- Projection update rule: public projection state advances only by replacing a revisioned snapshot. Partial public projection APIs that update active navigation, title, media information, operation availability, or active zoom independently are backdoors and must be removed after their replacement tests exist.
- Compatibility boundary rule: once a durable public value has a named owner, production facades must not keep public mutators, writable QML properties, direct event-filter dispatch, or helper APIs that can bypass that owner. Test fixtures may drive an owner through a dedicated fixture port, but production C++ callers must use the same typed command or snapshot-fact API that QML uses.
- UI gate rule: UI-local action gates are facts, not policy. QML may report a revisioned gate snapshot for focus, modality, active or hovered view, fullscreen chrome, and transient tools. The action runtime may ignore stale gate snapshots and must never recompute shared action availability from unrevisioned QML booleans.
- Fixed shortcut rule: fixed viewer-only shortcuts are runtime shortcut routes. They are not user-configurable actions, but they share the action runtime's newest UI gate, modal suppression, focused-view selection, trigger serialization, and command dispatch path.

## Minimum Source Key Contract

The C++ location and session domains own source identity keys for route sealing, route equality, route freshness, session snapshots, and direct-media scope identity.

Top-level route keys normalize `QUrl` path segments, preserve query and fragment components, and compare fully encoded identity strings. They do not case-fold and do not resolve symlinks, aliases, shortcuts, or platform filesystem equivalences.

Local file keys additionally clean local path syntax and make relative local paths absolute for identity only. The public `sourceUrl` remains the requested URL rather than the identity URL.

Direct-media scope identity is `{ current key, parent key, generation }`. The current and parent keys use the existing navigation-source rules, including document-portal host paths and KIOFuse archive restoration, before applying source-key normalization.

Direct-media freshness changes only when the effective current key or parent scope key changes. Resolving pending direct-image confirmation to an equivalent displayed URL is a phase change within the same freshness generation.

Archive-entry key families, render keys, thumbnail cache keys, predecode candidate keys, and sandbox-specific freshness beyond the existing navigation-source handling are defined by the extension contracts. Code that crosses adapter, cache, thumbnail, predecode, or render boundaries should use typed family-specific keys instead of a generic URL identity key.

## Derived Public State

QML-facing values may be derived from multiple C++ runtime states. For example, a public loading or status property may combine document state with an active presentation transition. The derived value must not become a second mutable source of truth. Keep the canonical owners explicit, and make notification dependencies follow the derived value.

When a public value has mode-specific ownership, only the active mode owns that value. Inactive mode state is a cache, projection, or restoration point, not a competing owner. Transition code must synchronize the next active owner before exposing the mode change. For example, different presentation modes may use different internal state owners, but the public value must have exactly one active owner at a time.

Image presentation mode transitions publish an explicit phase: previous-active while the last committed presentation is still authoritative, transitioning-placeholder while no partial target presentation may be rendered, and committed-active after the new single-page or two-page spread presentation is complete. The active presentation owner holds the canonical zoom snapshot, rotation, viewport frame, visible source rect, spread mode, reading direction, primary and secondary visibility, and render snapshot projection for the committed mode. Preparing a secondary page may update loader-owned decoded surfaces, but it must not copy zoom into page controllers or expose a partially prepared spread as the active public owner. If a transition aborts, the owner restores from the retained restoration snapshot and leaves the previous committed presentation authoritative.

For tiled static images, the current render frame is the projection boundary between presentation state, tile decode scheduling, and scene graph upload. The frame is derived from one snapshot containing the displayed surface identity and revision, render surface generation, display size, visible item rect, device pixel ratio, rotation, and displayed page role. The same frame decides which active tile layer may be drawn and which missing tile keys should be decoded. QML supplies viewport and scroll inputs only; it must not compute active tile layers, SVG scale buckets, draw entries, missing tile requests, or stale-tile fallback policy. Render nodes consume projected draw entries and may synchronize GPU resources from entry identities, but they must not independently recompute cache selection policy from the surface. Window changes, QRhi replacement, scene-graph invalidation, or device-loss-equivalent events invalidate the render surface generation so pending tile decode completions and uploaded textures cannot be reused for the wrong surface.

The active navigation projection must clear or mark unknown values during empty startup, loading intervals without a confirmed same-scope selection, mode switches, and replacement failures before publishing any current/count pair. Unknown navigation disables entry and shared dispatch, and must not display stale current/count values from a previous document.

Shared Previous and Next requests return their dispatch or boundary outcome from the document-session owner. The action runtime owns visible placement availability for Previous and Next from the session active-navigation projection, including disabled first/last boundary placements, while shortcut and fixed viewer-navigation paths may still ask the session for boundary feedback. QML may turn a first/last boundary outcome into user-facing feedback, but it must not decide boundary-vs-dispatch from raw current/count or mode-specific document values.

The main window title subject is a document-session-owned derived public value. Leaf image and video documents may own file-name-only title inputs and intrinsic media metadata, and image documents own page-navigation state, but the session combines those inputs into the public title subject. QML may append the application name around that subject, but it must not recompute direct-media size, document page counters, or mode-specific title fallback policy from raw document properties.

Media information is a document-session-owned projection. Leaf image and video documents own raw displayed URL, readiness, opened-collection scope, dimensions, and embedded metadata snapshots, but the session chooses the current information target, title, summary, row groups, copy-path availability, and containing-folder availability from the same committed mixed-media input revision as the rest of the public session projection. `KiriMediaInformation` is a stable QML facade over that snapshot and owns only Qt model notification mechanics plus clipboard and file-manager side effects.

Direct media sibling discovery may complete asynchronously, but the cursor used for the eventual public readout is the session-owned requested direct media URL for that open request. It must not be derived from a stale or empty image-document displayed URL. The session-owned direct media scope snapshot contains the effective cursor URL, parent scope URL, and generation used to accept or reject sibling-list completions; pending-image-to-stable-image confirmation is a phase change, not a new scope, when those URLs are unchanged.

Action availability is a session-fed named projection consumed by the application action runtime. It may include image-internal, viewport, shortcut-gating, and presentation-only facts, but those facts must enter the action runtime as C++ snapshots rather than QML-computed booleans. Shared Previous, Next, First, and Last availability comes from the session active navigation projection.

Extracting active navigation helpers must not move authoritative state to Rust. Rust may later compute pure projections from plain snapshots, but C++ applies and publishes the result through the document session.

Direct media cursor generation is part of stale async-completion rejection. Future helper extraction must preserve idempotent cursor operations and reject candidate results unless the cursor URL, scope URL, and generation still match the accepted session request. Cursor generation changes when the effective direct media identity or parent scope changes, not merely when ownership transitions from pending to confirmed for the same direct media URL.

Mixed-media deletion fallback remains session-owned for ordinary direct media scopes. Archive and directory collection deletion remains image-document owned unless a later ADR changes that ownership.
