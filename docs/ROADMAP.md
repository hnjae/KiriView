# KiriView Provider Rendering Roadmap

This roadmap moves KiriView from an application-owned QRhi renderer with explicit tile, mip, and SVG bucket composition to a provider-backed Qt Quick image pipeline. The correct end state is not a cheaper render-node wrapper: presentation still owns logical image display policy, decode workers still own expensive image preparation, Qt Quick owns texture upload, filtering, mipmap use, scene graph rendering, and visual item lifecycle, animated-image presentation defaults to provider-backed revisions and may use an animation-only presenter only when it still delegates visual drawing to high-level Qt Quick image/provider items rather than custom scene graph or texture rendering, and KiriView no longer includes, links, probes, or tests production image display through QRhi. The migration intentionally treats Qt Quick's high-level `Image` and provider APIs as the preferred maintenance boundary even with unlimited engineering capacity: if one safe whole display image cannot represent a demand, KiriView reports bounded-detail, unsupported, or failed state instead of rebuilding a custom visual tile presenter.

Each stage or named substage is sized so one Codex session can plausibly finish it, including focused tests and a commit. Larger stages list required split points that should be implemented as separate commits or sessions without changing the overall stage order. Follow the repository intent order for every behavior or structure change: spec commit when user-visible behavior changes, architecture commit when ownership or package structure changes, failing test commit where practical, then implementation commit. Stage-local test lists describe focused iteration checks and required new coverage; they do not replace the repository's final verification rules for code changes, source manifests, dependency wiring, or cross-language integration.

User-facing specs must not get ahead of visible behavior. Architecture docs and ADRs may declare the provider-rendering target early, but `docs/spec/` must describe either behavior already visible in the current implementation or behavior made visible by the immediately following implementation layer. Shadow provider projections, thumbnail previews that are not actually shown, provider-backed animation, bounded-detail states, and QRhi removal must not be advertised in user-facing specs until the stage that makes them externally observable.

**Diagnostic anchors**

- Current visible static-image display flows through `src/facade/kiriimageview.cpp`, `src/rendering/kiriimagerendernode.cpp`, render-frame projection, tile decode scheduling, decoded tile cache, custom shaders, and direct QRhi resource management.
- Current predecode caches `StaticImagePayload`, which includes an `ImageTileSource` plus a preview `QImage`; the target display hit should instead promote a ready display `QImage` into a provider store so the selected image appears immediately without constructing tile surfaces.
- Current dedicated first-display paths are valuable and should survive as fallback initial previews: `firstDisplayDecodeContext()` carries physical viewport size into decode requests, `QImageReaderTileSource::decodeFirstDisplayImage()` uses scaled JPEG decode, and `SvgTileSource::decodeFirstDisplayImage()` renders a viewport-sized SVG raster quickly.
- Most static raster formats do not have a dedicated cheap first-display path today; they fall back to blocking bounded preview decode, which makes valid existing desktop thumbnails useful as a generic preview tier before decode or refinement catches up.
- Current tile scheduling uses first-display hints to defer detailed tile decode until zoom needs more detail; provider rendering replaces that with a first provider image followed by stale-safe whole-image refinement buckets.
- Current SVG support correctly keeps parsing and rasterization in Rust through `resvg`, but its visible quality path is tied to SVG tile buckets; provider rendering should keep `resvg` and replace SVG tile composition with whole-surface SVG raster buckets.
- Current RAW decode already depends on LibRaw for full processing; provider rendering may use LibRaw `unpack_thumb()` as a RAW-only embedded preview source before full RAW processing, without adding generic EXIF, HEIF, JXL, or WebP embedded-thumbnail extraction.
- The existing thumbnail provider and store in `src/session/thumbnailimagestore.*` prove the local `image://...` pattern, but main image display needs page role, revision, priority, source identity, DPR, display bucket, quality state, and stale-completion ownership.
- Current active-navigation thumbnail lookup already validates and displays generated thumbnail images; the main image path can reuse that idea to consume existing XDG thumbnails as preview-only display seeds without making thumbnail generation part of the critical load path.
- Qt Quick `Image.sourceSize` and provider `requestedSize` are useful final-size inputs, but slow decode, SVG rasterization, file I/O, and stale workflow decisions must not happen inside an ordinary provider request.
- APNG and other animated image paths already display an initial `QImage` frame before playback continues; provider rendering should route each accepted frame through the same display-source projection and acceptance rules as still images instead of rebuilding QRhi frame surfaces.

## Target Architecture

Still-image rendering is provider-backed. KiriView registers a main display provider such as `image://kiriview-images/<entry-id>`, backed by a thread-safe `DisplayImageStore`. The provider is cache-only, reentrant, and cheap: `requestImage()` returns an already-published `QImage`, sets the original image size, may honor a valid `requestedSize` only for bounded final scaling of the stored image, and never performs source decode, SVG rasterization, file I/O, cache policy decisions, stale-completion acceptance, or public state mutation. When honoring `requestedSize`, the provider preserves aspect ratio, downscales only, accepts width-only or height-only requests by deriving the missing dimension proportionally, ignores requests with no positive dimension, invalid negative dimensions, oversized dimensions, or no-op sizes, and still reports the original image size through the `size` out parameter. `requestedSize` is not a decode, refinement, or SVG render demand signal; expensive size changes publish a new accepted display entry instead. The ordinary provider path must remain safe under Qt Quick's image-loading threading and caching rules: it is reentrant, uses revision-unique provider URLs or disabled QML image caching for dynamic display entries, and does not depend on `requestImage()` being called for every bound URL. If a future miss path needs background work, it uses `QQuickAsyncImageProvider` and `QQuickImageResponse` with provider-owned cancellation; ordinary `QQuickImageProvider` misses return an empty image or explicit missing state.

Presentation remains the logical owner of image display state. `ImagePresentationRuntime` owns zoom, rotation, viewport frame, spread mode, reading direction, page visibility, display quality state, and the active display-source projection. It does not own QRhi resources, tile draw entries, texture lifetimes, or scene graph nodes. For each displayed page role, presentation publishes provider URL, source revision token, source identity and selected-source scope, original image size, displayed raster size, accepted provider `sourceSize` hint if one is needed for the current entry, quality state, load-acknowledgment requirement, missing/error/unsupported status, same-source retention and `retainWhileLoading` eligibility, visible item rect, page geometry, rotation, and whether the current image is exact, first-display, thumbnail-preview, refinement-pending, stale-retained, bounded-detail, unsupported, or failed.

Qt Quick owns visual image item rendering. QML keeps the existing viewport and spread item geometry, but the still-image page visual becomes a small provider-backed `Image` component instead of `KiriImageView`. Animation frames should use the provider-backed `Image` revision path by default; any measured exception must still be an animation-only Qt Quick composition that delegates drawing to high-level image/provider items, not a custom visual `QQuickItem`, scene graph node, texture provider, framebuffer path, shader, or direct texture owner. QML binds `source`, item size, position, visibility, rotation, `smooth`, `mipmap`, `cache`, `asynchronous`, `retainWhileLoading` only when the accepted projection explicitly allows same-source retention and the Qt version supports it, and `sourceSize` only when the accepted display entry owns that hint. The still-image presenter chooses an explicit `Image.asynchronous` policy: provider-ready cache hits may load synchronously when that best supports deterministic load acknowledgments, while any asynchronous path must still use source URL plus revision acknowledgments for release safety. `retainWhileLoading` must be disabled or cleared across source-identity changes so a failed or loading replacement cannot visually retain a previous media target. `sourceSize` changes reload a QML `Image`, so pan, zoom, DPR, or viewport demand changes must not churn `sourceSize` for the same provider URL; visual scaling between accepted buckets uses item geometry, and expensive bucket changes publish a new accepted provider entry. The main display normally disables QML engine image caching because `DisplayImageStore` owns dynamic display lifetime; if a measured path enables engine caching, provider URLs must still be revision-unique so stale cached `image://` results cannot mask a newer accepted display entry. QML does not choose decode buckets, SVG scale buckets, cache retention, provider ids, animation timing, stale acceptance, or fallback quality policy.

Render-context discovery survives without owning rendering or QRhi. A non-rendering Qt Quick viewport context bridge supplies display DPR, item attachment, scene-graph invalidation, generation changes, and a resolved display texture cap to the presentation/refinement owner. The display texture cap comes from stable public Qt/Qt Quick capability APIs when available, otherwise from a conservative configured fallback; it must not call `QQuickWindow::rhi()`, include `<rhi/qrhi.h>`, link `Qt6::GuiPrivate` for RHI access, or depend on QRhi resource limits. `KiriImageView` no longer owns this discovery after the QML image swap, and secondary page visuals must not install competing document render-context providers.

Displayed image lifetime is centralized in a display-image store. Page-surface controllers own display entries for their current role, including primary, secondary, and animation-frame entries. They publish immutable display entries, reprioritize visible and stale-retained entries, hold explicit pin leases for visible, stale-retained, pending-load, and previous-frame-retention entries, release superseded entries only after display policy and load acknowledgments no longer reference them, and report revision changes to presentation. Display entries carry a stable entry id, original source identity, original image size, raster size, byte cost, display quality, preview origin, page role or animation role, source generation, and optional debug label. Entry ids are never reused during a process lifetime, and every newly accepted display entry with different image bytes publishes a fresh provider URL; status-only stale-retained revisions keep the previous URL and do not allocate a new pending-load lease. The store's byte budget is resolved with the image cache budgets and replaces the old visible static tile cache budget by the end of the migration; temporary compatibility code may share or mirror the old `staticTileCacheByteBudget`, but Stage 11 must remove or intentionally rename tile-era budget vocabulary. Pending-load leases are released by a matching loaded, error, or explicit missing acknowledgment for the same provider URL and revision, or by supersession, clear, or display-policy cancellation; provider misses must not leave entries pinned indefinitely. Stale retention is limited to the current selected source scope and display-source owner, such as refinement or bucket replacement handoff; selecting a different media target still clears the previous image presentation unless the new target is immediately available from predecode.

Fast first display is a first-class decode result, but not a mandatory stage after thumbnail preview. JPEG, SVG, and any decoder that supports cheap scaled first display may produce a viewport/DPR-sized `QImage` only when no matching predecode payload and no valid thumbnail preview were accepted for the load. That first image is immediately accepted through the normal image-load operation identity and published to the provider store. A separate refinement demand then asks for the next appropriate whole-image bucket. Stale refinement completion cannot replace a newer source, page role, zoom or rotation generation when rotation affects bucket selection, DPR generation, render-context or texture-capability generation, memory-budget or allocation-policy generation or resolved cap, or display-source revision.

Thumbnail preview is a generic preview fallback tier for local still-image sources, especially formats without a dedicated fast first-display decoder. After a predecode miss, KiriView chooses one initial preview path: a valid XDG thumbnail cache entry wins; if no XDG thumbnail is accepted and the source is RAW, a LibRaw embedded preview may run; if no thumbnail preview is accepted, a dedicated decoder first-display path may run; otherwise normal bounded decode or refinement begins. When a local file source has a valid XDG thumbnail cache entry, KiriView may publish the best available `xx-large`, `x-large`, or `large` thumbnail as a provider entry with `thumbnail-preview` quality and `xdg-thumbnail` origin before source decode finishes. Thumbnail preview acceptance verifies source URI, freshness metadata, trusted original image size after image-reader orientation or equivalent post-transform metadata has been applied, and preview aspect compatibility; it never marks the image exact, skips dedicated JPEG/SVG first-display for that source load, never blocks on thumbnail generation, and is replaced by normal decoded, predecoded, or refined display payloads as soon as they are accepted. A preview without trustworthy post-transform original dimensions is rejected or kept out of the visible display-source projection so Fit, 100% zoom, scroll bounds, and aspect-preserving placement do not use thumbnail dimensions or unrotated source dimensions as source dimensions. KIO or platform thumbnail generation may run only as optional background preparation for future opens, not as the critical first-display path.

RAW embedded preview is the only embedded-file thumbnail path in the end state. For RAW formats handled by LibRaw, KiriView may call `unpack_thumb()` after an XDG thumbnail miss and before full RAW processing. If LibRaw returns a JPEG preview, Qt may decode those preview bytes into a display `QImage`; if it returns an unpacked bitmap preview, KiriView may convert it to the same display image contract. RAW embedded preview uses `thumbnail-preview` quality with `raw-embedded-thumbnail` origin, is never exact, is never used for non-RAW still images, and is silently skipped when missing, invalid, too small, too large, unsupported, or slower than the load generation that requested it. Generic embedded previews for JPEG EXIF, HEIF thumbnail items, JXL codestream previews, WebP chunks, and similar formats stay out of scope unless Qt later exposes them through a stable image-reader API.

Thumbnail preview is still-image-only. Animated image files do not use external thumbnail preview entries, even as posters; they wait for decoded first-frame publication and then continue through the animation-frame display-source path. Video-frame predecode remains out of scope.

Predecode caches provider-ready display images. Direct-media and image-document predecode store display `QImage` payloads with location, opened-collection scope, source identity, original image size, display raster size, quality state, metadata, and byte cost. A matching predecode hit promotes the cached image into the displayed provider store without another decode. Predecode may also schedule refinement for nearby images, but it remains still-image-only and never attempts video-frame predecode.

Raster image refinement uses coarse whole-image buckets instead of app-owned tiles or mip pyramids. Bucket selection consumes original image size, display size, visible viewport, zoom, DPR, rotation, memory budget, and the resolved display texture cap. Bucket admission carries render-context or texture-capability generation plus memory-budget or allocation-policy generation, or an equivalent resolved byte/pixel cap, so stale completions cannot bypass newer limit decisions. Qt image readers, HEIF, RAW, and other static decoders return complete display `QImage` buckets. Decoder implementations may use source-internal tiling, such as HEIF grid tile decode, only to assemble one accepted display bucket; they must not publish multiple visual page tiles, own viewport tile scheduling, or reintroduce a provider-era tile renderer. If no safe initial display bucket can be produced, KiriView publishes an unsupported or failed display state for the selected target. If a lower-detail accepted bucket is already usable but the requested refinement would exceed memory, size, or texture limits, KiriView keeps that bucket visible with explicit bounded-detail state instead of silently restoring custom tile rendering.

SVG remains Rust-rendered and resolution-independent at the source layer. KiriView keeps `usvg`, `resvg`, and `tiny-skia` as the supported static SVG renderer, keeps SVG source bytes and intrinsic size as the durable source payload, and renders whole-surface SVG raster buckets keyed by source identity, intrinsic size, target display scale bucket, DPR, render-context or texture-capability generation, memory-budget or allocation-policy generation or resolved raster cap, rotation generation when rotation affects bucket selection, and render revision. SVG does not use QtSvg, app-owned SVG tiles, or per-pan `sourceSize` churn. The current provider image remains visible until a newer SVG bucket is accepted, then QML receives a fresh provider URL; revision tokens are metadata for stale rejection and acknowledgments, not a substitute for changing the bound `Image.source`.

Animation frames use the display-source projection path. The animation player remains the owner of timing, loop, cancellation, and frame acceptance. Each accepted frame is decoded or composited into a display-ready `QImage` and published through the same source-generation and page-role acceptance rules used by still images. The default presenter may expose accepted frames as provider-backed `Image` revisions, but frame timing must not depend on a slow or asynchronous provider request becoming visible; the frame entry must already be provider-ready, cache-disabled or revision-unique, and retained until an explicit load-outcome acknowledgment or fixed previous-frame retention lease says the old frame is safe to release. Synchronous image loading is an optimization only, not the release-safety proof. If provider source churn proves unsuitable for high-rate sequences, KiriView may use a narrow animation-only Qt Quick presenter only as the documented exception that still consumes accepted display entries through high-level image/provider items and may degrade frame cadence or report unsupported playback instead of introducing custom visual rendering. That presenter must not own direct QRhi resources, custom `QQuickItem` rendering, `updatePaintNode()`, custom scene graph nodes, app-managed `QSGTexture` lifetimes, `QSGTextureProvider` plumbing, framebuffer paths, custom shaders, decode work, timing policy, cache policy, stale acceptance, tile scheduling, or general image rendering strategy.

Failure and quality states are explicit. Unsupported decode, too-large display buckets, failed refinement, missing display entries, failed SVG rasterization, stale thumbnail previews, and stale completions are represented as owned result states. QML displays the current accepted display image when same-source policy allows stale retention, otherwise it displays the normal loading or error state. A lower-quality first-display or thumbnail-preview image must not masquerade as exact detail after a newer accepted quality decision says it is insufficient. Any new user-facing text for these states must enter the localization flow when the state becomes visible, including template updates and the appropriate i18n checks.

The old QRhi image renderer and QRhi dependency are removed after migration. `KiriImageView`, `KiriImageRenderNode`, image shaders, render-frame draw entries, tile decode scheduler/runtime/state, decoded tile cache, `StaticTileSurface`, SVG tile bucket composition, render-frame public vocabulary, `Qt6::GuiPrivate` RHI linkage, `<rhi/qrhi.h>` includes, and app-owned image mip/tile tests disappear. Any remaining image helpers shrink to decoder-only display-payload preparation and public-Qt display capability projection. Boundary tests prevent reintroducing QRhi usage, custom image render nodes, or visual tile scheduling from facade, presentation, QML, decoder, or test paths.

## Stage 1: Declare Provider Rendering Intent

**Goal:** Land the provider-rendering architecture contract before touching the renderer, without making user-facing specs claim behavior that is still shadow-only or unimplemented.

**Work:**

- Audit `docs/spec/image-display.md` for tile-era language, but with the current stage order this should normally produce no Stage 1 spec edit because Stage 2 intentionally has no visible rendering change. Update specs in Stage 1 only if Stage 1 itself adds or removes externally visible behavior; otherwise defer user-facing spec edits to the first substage that makes the behavior visible. Do not describe shadow provider projections, thumbnail previews, RAW embedded previews, bounded-detail behavior, provider-backed animation-frame display, or QRhi removal as user-visible behavior until the stage that makes each one visible.
- Record target visible behavior in architecture docs, ADRs, or roadmap text when it is not yet externally visible: thumbnail preview, RAW embedded preview, fast first-display visibility, later sharpness refinement, same-source stale-retained images, no cross-source restoration on source-change failure, bounded-detail behavior, oversized-image errors, animation-frame display, and SVG sharpness readiness.
- Rewrite production-path architecture docs so they no longer describe active QRhi image rendering, tile scheduling, render-frame composition, or tile-source contracts as the target production path. Keep current-state inventories explicitly labeled as current until implementation catches up, and put provider-rendering rules in target-direction, ownership, or ADR text so architecture docs do not claim that unimplemented code already exists. State that presentation owns display-source projection, `DisplayImageStore` owns provider-entry lifetime, decode workers own expensive image preparation, refinement jobs own whole-image bucket production, and Qt Quick owns texture upload/filtering/rendering.
- Record provider constraints in architecture docs: ordinary provider requests are cache-only and reentrant, provider ids are generated by runtime owners, QML engine caching cannot be used as a freshness mechanism, `Image.asynchronous` policy belongs to the accepted presenter path, display-store byte budgets replace tile-cache display budgets by the end state, and slow work belongs to decode/refinement jobs or an explicit async image provider.
- Add only boundary tests that can be made meaningful within the next implementation stage and compile against existing abstractions. Do not add provider-specific failing tests in Stage 1 unless they compile without the provider foundation; add provider request-boundary red tests in Stage 2 immediately before implementing `DisplayImageProvider`. Do not add long-lived failing tests for QRhi removal, QML renderer removal, or tile-scheduler deletion in this stage; reserve those end-state guards for Stages 11 and 12.
- Commit the spec, architecture, and any failing-test layers separately according to the intent order.

## Stage 2: Add The Display Image Provider Foundation

**Goal:** Add provider infrastructure without changing visible rendering yet.

**Work:**

- Add `DisplayImageStore`, `DisplayImageProvider`, `DisplayImageRetentionPriority`, and helper functions for `image://kiriview-images/<entry-id>` URLs, modeled after the thumbnail provider but in a rendering/display-owned module.

Provider entry ids are immutable and never reused.
- Define how the display-store byte budget is resolved from `ImageCacheBudgets`, how it coexists with the temporary static tile cache budget during compatibility stages, and what name replaces tile-era display-cache vocabulary by Stage 11.

- Register the provider in `src/application/applicationruntime.cpp` beside the thumbnail provider.
- Add source manifests, CMake/test target entries, and QML-engine registration wiring needed for the new provider module.
- Keep the provider cache-only: provider miss returns an empty image; valid requested sizes are handled as downscale-only, aspect-preserving, no-upscale returns with the original size still reported; width-only and height-only requested sizes derive the missing dimension proportionally; and request handling does not call document, presentation, decoder, or predecode owners.

## Stage 3: Introduce Provider-Ready Display Payloads

**Goal:** Split display-ready images from tile-source payloads while preserving current renderer compatibility.

**Work:**

- Introduce a display payload type for static decode results containing source identity, applied image-reader transform metadata, post-transform original image size, current display `QImage` in display orientation, display quality state, display pixels per source pixel, metadata, and an optional refinement-capable source payload.
- Preserve the current JPEG fast path conceptually as a selectable initial preview payload: `firstDisplayDecodeContext()` still uses physical viewport size, and JPEG still performs scaled first-display decode when no predecode or thumbnail preview already satisfied the initial display.
- Change static decode planning so a selected first-display `QImage` can be accepted as a valid display payload even when a higher-detail refinement is still pending.
- Distinguish hard decode errors from optional first-display unavailability. A decoder-owned first-display error for the selected source may still fail the load, but skipped, unsupported, oversized, or stale optional preview attempts must fall through to normal decode or refinement when that is still valid.
- Keep a temporary compatibility adapter that can still create the old `StaticImagePayload` or surface path for QRhi rendering until QML is switched.
- Add tests proving JPEG first-display decode produces a smaller provider-ready payload, PNG and small JPEG fall back to blocking display payloads when appropriate, hard first-display errors remain load errors, optional preview unavailability falls back, metadata survives the new payload shape, and EXIF/image-reader auto-transform produces the expected display orientation and post-transform logical size.
- Add tests proving first-display quality is represented as first-display/refinement-pending rather than exact detail.
- Commit the payload model before changing presentation projections.

## Stage 5: Add Thumbnail Preview Tier

**Goal:** Publish valid existing XDG thumbnails and RAW embedded previews as preview-only provider display sources while normal decode and refinement continue. They remain shadow display-source projections until the later QML provider-image swap unless the legacy compatibility renderer is driven from the same accepted display entry.

**Work:**

- Add an external thumbnail preview owner for ordinary local still-image sources that checks XDG thumbnail cache entries in `xx-large`, `x-large`, then `large` order, reusing the existing thumbnail cache lookup bridge where practical instead of inventing a second XDG metadata parser.
- Add a RAW embedded preview owner that uses LibRaw `unpack_thumb()` only for RAW sources after an XDG thumbnail miss.
- Choose exactly one initial preview path after a predecode miss: valid XDG thumbnail preview first, RAW embedded preview second for RAW sources only, dedicated decoder first-display third, and the existing normal bounded/static decode path fourth. Any future refinement demand is recorded for Stage 7 instead of executed here.
- Split this stage into at least four substages: Stage 5a adds XDG cache lookup and metadata validation without publication, Stage 5b publishes and replaces XDG thumbnail preview entries, Stage 5c adds RAW embedded preview extraction and validation after XDG miss, and Stage 5d publishes and replaces RAW embedded preview entries. Each substage should have focused tests and its own commit.

- Verify thumbnail identity and freshness against `Thumb::URI`, `Thumb::MTime`, and `Thumb::Size`, and verify post-transform original dimensions against trusted image-size and orientation metadata or a cheap source-header probe that applies image-reader orientation semantics; reject stale, missing, dimensionless, unrotated-dimension-only, aspect-mismatched, or otherwise mismatched entries.
